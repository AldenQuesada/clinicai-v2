P-12 · Presence / typing Fase 3+4

Env crítica identificada Supabase Realtime 10 commits no módulo

Fases 3 e 4 do P-12 dependem de Supabase Realtime para Presence (quem está online) e typing indicators. O hook usePresence e os componentes PresenceAvatars / PresenceLine consomem o canal broadcast do Supabase via NEXT_PUBLIC_SUPABASE_* — variáveis que precisam estar disponíveis no contexto de BUILD do Easypanel (não só em runtime). O fix 87f2fde evidencia que esta configuração estava ausente ou incorreta.

Envs obrigatórias · Easypanel → Lara → Build Args
Variável Onde configurar Propósito Status
NEXT_PUBLIC_SUPABASE_URL Easypanel → App lara → Environment & Build Args → Build Args URL do projeto Supabase. Necessária em build para que o cliente browser seja compilado corretamente pelo Next.js. Verificar
NEXT_PUBLIC_SUPABASE_ANON_KEY Easypanel → App lara → Environment & Build Args → Build Args Chave anon do Supabase. Usada pelo Realtime para autenticar canais de Presence e typing no browser. Verificar
Por que Build Args e não só Runtime Env? Variáveis prefixadas com NEXT_PUBLIC_ são injetadas pelo Next.js em build time nos bundles JS do cliente. Se estiverem ausentes no build, o valor será undefined mesmo que a env exista em runtime — quebrando silenciosamente o Supabase client no browser. O fix 87f2fde contornou isso injetando a config em runtime via variável de servidor, mas o caminho canônico é garantir os Build Args no Easypanel.
