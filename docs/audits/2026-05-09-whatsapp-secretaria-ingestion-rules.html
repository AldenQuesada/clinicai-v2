⚖ 10 Regras inegociáveis · pra arquitetura e patches futuros

Regras codificadas em 2026-05-09 após auditoria do dash WhatsApp Secretaria. Aplicar SEMPRE em trabalho que toque ingestão WhatsApp, fila ordenada, webhook Evolution/Cloud, wa_messages, wa_conversations, wa_chat_mirror, wa_webhook_log ou views derivadas.

REGRA 1
Nunca diagnosticar só por view agregada
Views (wa_webhook_event_audit_view, signature_reasons_seen) agregam stages com bool_or/string_agg por provider_msg_id. Misturam hits de momentos diferentes. Pra afirmar "parou cedo", auditar wa_webhook_log hit-por-hit.
REGRA 2
"Não encontrei por provider_msg_id" ≠ "perda real"
Match alternativo nessa ordem antes de declarar perda: provider_msg_id → wa_message_id → conversation_id+texto exato → texto parcial → janela de horário → deleted_at.
REGRA 3
Patch não generaliza
Documentar: escopo coberto, escopo não coberto, smoke feito, smoke pendente, risco residual. "Patch A resolveu outbound externo @lid + conversa existente" ≠ "Patch A resolveu cold outreach LID sem conversa".
REGRA 4
Antes de tabela nova, auditar schema existente
Rodar information_schema.tables/columns + pg_constraint + pg_indexes. Muitas vezes existe estrutura próxima vazia (caso wa_contact_identities). Propor "popular/usar/hardening" ≠ "criar nova tabela".
REGRA 5
Nunca misturar sentinelas de cenários diferentes
Cada caso carrega: bucket, hipótese, evidência, próxima query, critério PASS. Errado: lista única "Ana, Arildo, Evanir, Gilmar". Certo: separar por bucket (parada precoce / after_saveInbound_ok sem match / cold outreach LID / fila stale).
REGRA 6
PASS_DB ≠ PASS_API ≠ PASS_UI ≠ PASS_E2E
Banco/SQL prova banco. curl prova API. UI exige Playwright/Puppeteer ou validação manual. NUNCA declarar "UI funciona" provando só DB.
REGRA 7
sent_at canônico só pós-Patch Timestamp A
Pós-c5dfc7f: sent_at = data.messageTimestamp (real WhatsApp); created_at = gravação no DB.
Pré-c5dfc7f: ambos ≈ hora do webhook · sent_at contaminado · não usar como verdade. Sempre delimitar período.
REGRA 8
Backfill com regras estritas
sent_at de messageTimestamp, não hit_at. last_message_at só atualiza se sent_at_new > current. UNIQUE(clinic_id, provider_msg_id). Não subir conversa por created_at. Redelivery antigo não substitui last se sent_at antigo.
REGRA 9
Trace técnico ≠ estado operacional
signature_reason é útil mas não basta. Pipeline precisa terminal state explícito: persisted | duplicate | skipped_* | pending_* | failed_*. auth_ok + event_messages_upsert SEM terminal é anti-pattern.
REGRA 10
Classificar causa antes de escolher patch
Causas: perda real / ID divergente / soft-delete / evento especial / número interno / grupo / fila stale / API filtrando / UI cache / redelivery. Cada causa = patch diferente. Patch sem classificação = chute.

📄 PARTE I · WhatsApp Secretaria Ingestion Audit, Ordered Queue Audit & Architecture Notes

Documento original · 2026-05-09 · canal Mih/Secretaria · provider Evolution · escopo: mensagens que chegam no celular mas nem sempre entram no dash, divergências na fila ordenada

1. Resumo executivo

Durante a auditoria geral pós-patches do WhatsApp Dash da Secretaria, foi confirmado que o sistema avançou em três frentes importantes:

  1. Patch A corrigiu outbound externo fromMe=true + @lid quando já existe conversa no Mih.
  2. Recovery manual da Evanir foi executado com sucesso.
  3. Patch Timestamp A corrigiu wa_messages.sent_at para usar o horário canônico do WhatsApp vindo de raw_body.data.messageTimestamp, deixando created_at como horário de gravação/processamento.

Depois dessas correções, a auditoria geral revelou um problema mais profundo:

Ainda existem eventos reais inbound fromMe=false + @lid que chegam ao webhook Evolution, mas não entram no pipeline completo de persistência.

Em alguns casos, o evento para logo após:

evo:auth_ok
evo:event_messages_upsert

sem passar por:

evo:phone_resolved
evo:tenant_resolved
evo:lead_conv_resolved
evo:before_saveInbound
evo:after_saveInbound_ok

Isso explica por que o usuário vê mensagens no WhatsApp/celular, mas elas não aparecem no dash.

Além disso, existe uma segunda frente relacionada, que precisa entrar na documentação e na estratégia: auditoria da fila ordenada das conversas.

O problema operacional não é apenas "a mensagem foi salva ou não foi salva". Também existe risco de a conversa existir, mas aparecer na posição errada, com last_message_at errado, last_message_text desatualizado, cache/API/UI fora de sincronia ou ordenação diferente da lista real do WhatsApp/celular.

Portanto, a frente correta é dupla:

  • A. Ingestão: garantir que toda mensagem real vire wa_message ou pendência auditável.
  • B. Fila ordenada: garantir que a lista da Secretaria reflita a ordem real de atividade das conversas.
Regra central:
Nenhum webhook real de mensagem pode desaparecer silenciosamente.
Nenhuma conversa com mensagem nova pode ficar fora da posição correta da fila operacional.

2.3 Patch Timestamp A — Evolution

PASS em produção

Problema anterior:

O webhook Evolution usava horário de processamento como sent_at. Isso causava problemas de ordenação, histórico e last_message_at, principalmente em redelivery/backfill.

Correção:

sent_at    = raw_body.data.messageTimestamp
created_at = horário real de gravação/processamento

Smoke validado:

Conclusão: wa_messages.sent_at agora representa o horário real do WhatsApp · wa_messages.created_at representa o horário de gravação no banco. Essa correção também é fundamental para a fila ordenada das conversas.

6. Auditoria da fila ordenada das conversas

6.1 Por que essa auditoria precisa entrar

A investigação não pode olhar somente para wa_messages. Mesmo quando a mensagem existe, o dash pode falhar operacionalmente se a conversa:

  1. Não sobe para o topo.
  2. Sobe com atraso.
  3. Fica atrás de conversas antigas.
  4. Mostra last_message_text antigo.
  5. Mostra last_message_at baseado em created_at errado.
  6. Está correta no banco, mas errada na API.
  7. Está correta na API, mas errada na UI por cache/SSE/paginação.
  8. Está correta no WhatsApp oficial, mas divergente do mirror/operational_view.

6.2 Fontes que precisam ser comparadas

FonteFunção
A. WhatsApp nativo/celularVerdade operacional percebida pela equipe.
B. wa_webhook_logEventos recebidos e stages do webhook · saber se evento chegou.
C. wa_messagesFonte canônica das mensagens persistidas.
D. wa_conversationsGuarda last_message_at e last_message_text usados para fila/lista.
E. wa_chat_mirrorEspelho do chat vindo da Evolution. Pode ter LIDs/conversas presentes no provider mas ausentes no dash.
F. wa_conversations_operational_viewView operacional Secretaria/KPIs/lista. Tem filtros, owners, status, deleted_at.
G. API /api/secretaria/conversationsPode paginar, filtrar e ordenar diferente da view. Pode usar cache.
H. UI /secretariaEstado local, paginação, SSE, cache, loadMore, filtros, tabs.

6.3 Regra correta de ordenação

A fila operacional da Secretaria deve ordenar conversas por atividade real, não por processamento.

Regra preferida:

ordered_at = greatest(
  wa_conversations.last_message_at,
  max(wa_messages.sent_at) por conversa
)

Se houver divergência, max(wa_messages.sent_at) deve ser usado para detectar se wa_conversations.last_message_at está stale.

Não usar created_at como fonte primária de ordenação.

6.4 Problemas possíveis na fila ordenada

  1. Conversa com mensagem nova em wa_messages, mas last_message_at antigo em wa_conversations.
  2. Conversa com last_message_at novo, mas sem wa_message correspondente.
  3. Conversa com last_message_text diferente da última mensagem real.
  4. Conversa com mensagem recebida no webhook, mas sem wa_message.
  5. Conversa no mirror, mas sem wa_conversation.
  6. Conversa no mirror/WhatsApp oficial, mas fora da lista da Secretaria.
  7. Conversa correta na view, mas não retorna na API por paginação/filtro.
  8. Conversa correta na API, mas não aparece na UI por cache/SSE/paginação.
  9. Conversa aparece em bucket errado: Secretaria/Mirian/Alden/Luciana/sem owner.
  10. Conversa active/paused/deleted filtrada de forma incorreta.
  11. Ordem afetada por last_message_at gravado com now() em vez de messageTimestamp.
  12. Ordem afetada por backfill antigo com sent_at errado.
  13. Mensagem inbound/outbound salva, mas updateLastMessage não rodou.
  14. Redelivery antigo subindo conversa indevidamente se usar created_at.
  15. Mensagem nova não sobe porque last_message_at não foi atualizado.

6.6 Auditoria da fila vs WhatsApp oficial

Quando o usuário fornece uma lista do WhatsApp/celular, a comparação correta deve usar:

  1. Número/phone quando existir.
  2. remote_jid/@lid quando não houver phone.
  3. display_name apenas como apoio, nunca como chave canônica.
  4. Último texto conhecido.
  5. Horário aproximado.
  6. provider_msg_id se disponível.

Regra: Match por número > match por remote_jid > match por texto/horário > match por nome.

Ordem de diagnóstico: wa_webhook_log → wa_messages → wa_conversations → operational_view → API → UI. Nunca começar pela UI se a mensagem nem existe em wa_messages.

6.9 Critério de PASS da fila ordenada

  • [ ] Top N da Secretaria no banco bate com Top N esperado por last_message_at.
  • [ ] Top N da API bate com Top N do banco.
  • [ ] Top N da UI bate com Top N da API.
  • [ ] Conversa com nova mensagem inbound sobe sem refresh manual.
  • [ ] Conversa com nova mensagem outbound externa sobe sem refresh manual.
  • [ ] last_message_at usa sent_at canônico.

  • [ ] Redelivery antigo não sobe conversa indevidamente.

8. Diagnóstico arquitetural

O problema raiz não é apenas um bug pontual. O desenho atual segue o padrão:

webhook recebe evento
→ tenta normalizar
→ tenta resolver telefone/tenant/conversa
→ tenta salvar
→ registra traces

Esse modelo é frágil porque depende de a execução inteira dar certo na mesma passada. Quando algo falha no meio, o evento pode terminar apenas em logs, sem uma entidade operacional durável para reprocessamento.

O padrão correto seria:

webhook recebe evento
→ grava evento bruto/normalizado em Event Inbox durável
→ state machine processa
→ evento termina em estado final auditável
Regra de ouro:
Nenhum evento real de conversa pode terminar sem estado final auditável.
Nenhuma conversa com atividade real pode ficar fora da ordem operacional correta.

9. Como poderia ter sido evitado

9.1 Event Inbox obrigatório

Todo webhook recebido deveria virar uma linha durável antes de qualquer lógica de negócio.

Estados possíveis: received | normalized | identity_resolved | conversation_resolved | persisted | skipped_group | skipped_internal | skipped_non_message | pending_identity | pending_conversation | failed_normalization | failed_resolution | failed_persistence | duplicate

Benefício: mesmo se phone_resolved falhar, o evento não some. Fica pendente e reprocessável.

9.2 State machine de ingestão

received
  → normalized → classified → resolved_identity → resolved_tenant
  → resolved_conversation → persisted_message

OU: received → skipped_group
OU: received → pending_identity
OU: received → failed_persistence

O que não pode existir: auth_ok/event_messages_upsert e depois nada operacional.

9.3 Identidade WhatsApp como entidade própria

Tabela wa_contact_identities + wa_identity_links com campos: phone_e164, lid_id, wa_id, sender_pn, remote_jid, participant_jid, instance_id, wa_number_id, confidence, source, first_seen_at, last_seen_at. Um contato pode ter múltiplas identidades. Uma identidade pode ser resolvida com confiança alta, média ou pendente.

9.4 Pending queues / DLQ

Eventos que não podem ser resolvidos não devem ser descartados. Devem cair em filas operacionais: wa_pending_identity_events · wa_lid_pending_outbound · wa_inbound_pending_resolution · wa_failed_persistence_events.

9.5 Reprocessamento idempotente

Depois de corrigir um bug, o sistema deveria permitir reprocessar eventos pendentes dos últimos X dias / por provider_msg_id / por remote_jid / por wa_number_id. Garantias: não duplicar wa_messages, não alterar mensagens já corretas, não criar conversa fake, não sobrescrever last_message_at errado, usar messageTimestamp como sent_at.

9.6 Ledger / auditoria de ingestão

Cada provider_msg_id deveria ter trilha: received → auth_ok → normalized → identity_resolved → tenant_resolved → conversation_resolved → message_persisted → conversation_last_message_updated. Ou estado final: skipped_* | pending_* | failed_* | duplicate.

Trace técnico não substitui estado operacional.

9.7 Métricas automáticas

Alertas: se event_messages_upsert > 0 e nenhum terminal state após N minutos · se fromMe=false + conversation + @lid + conversa existente e não persistiu · se after_saveInbound_ok e não existe wa_message associada · se max(wa_messages.sent_at) > wa_conversations.last_message_at · se API top N diverge do banco top N · se UI não atualiza após nova mensagem.

10. Arquitetura alvo recomendada

10.1 Fluxo ideal

Evolution Webhook ↓ wa_event_inbox.insert(raw event) ↓ normalize_event() ↓ classify_event() ↓ resolve_identity() ↓ resolve_tenant() ↓ resolve_conversation() ↓ persist_message() ↓ update_conversation_last_message() ↓ mark_event_persisted()

10.2 Estados finais obrigatórios

Obrigatório: persisted | duplicate | skipped_group | skipped_internal | skipped_non_message | skipped_unsupported_media | pending_identity | pending_conversation | failed_normalization | failed_resolution | failed_persistence

  • Para lista operacional: conversation_order_at = wa_conversations.last_message_at.

  4. Só então criar patch.

13. Posição operacional

O sistema não deve ser tratado como "quase ok" enquanto houver eventos reais de conversa sem estado final.

A regra operacional deve ser:

  • Banco manda.
  • Webhook recebido precisa virar evento auditável.
  • Mensagem real precisa virar wa_message ou pendência explícita.
  • Conversa com atividade nova precisa subir corretamente na fila.
  • Nada pode desaparecer silenciosamente.

14. Checklist para considerar a frente fechada

  • [ ] Todos os eventos critical inbound classificados.
  • [ ] Eventos realmente ausentes recuperados/backfillados com idempotência.

  • [ ] wa_conversations.last_message_text bate com a última mensagem real.
  • [ ] Top N da operational_view bate com Top N canônico do banco.
  • [ ] Top N da API bate com Top N da operational_view.
  • [ ] Top N da UI bate com Top N da API.
  • [ ] Mensagem nova inbound sobe conversa sem refresh manual.
  • [ ] Mensagem nova outbound externa sobe conversa sem refresh manual.

  • [ ] Não há conversa ativa sumida por filtro/paginação/cache.

15. Conclusão

O que foi descoberto não é apenas um bug visual do dash. É um problema de arquitetura de ingestão e fila operacional:

  • O handler atual tenta resolver e persistir direto.
  • Quando falha no meio, alguns eventos ficam apenas em logs técnicos.
  • Quando last_message_at/text não é atualizado corretamente, a conversa pode ficar fora da fila operacional correta.

A solução robusta é evoluir para: event inbox durável · state machine de ingestão · identidade WhatsApp robusta · pending queue · DLQ · reprocessamento idempotente · métricas de terminal state · auditoria de fila ordenada · contrato banco → view → API → UI.

Patches pontuais continuam úteis, mas a direção correta é:
Nenhum webhook de mensagem real deve terminar sem persistir, pular explicitamente ou ficar pendente de forma auditável.
Nenhuma conversa com atividade real deve ficar fora da posição correta da fila da Secretaria.

📄 PARTE II · Resposta Técnica + Regras de Treinamento

Auditoria do parecer Claude Opus 4.7 · validação de premissas · regras codificadas para a IA não repetir erros

0. Veredito geral

O documento da Parte I está correto na direção macro:

  1. O problema não é só "bug pontual".
  2. A frente correta é dupla: ingestão + fila ordenada.
  3. Patch técnico não substitui estado operacional.
  4. A arquitetura precisa impedir perda silenciosa.
  5. Não se deve fazer patch antes de classificar os cenários.

Mas estava certo em apontar riscos de premissa: tratar agregação como evento individual · chamar de "perda real" algo talvez salvo com outro ID · assumir created_at/sent_at universalmente · ignorar que wa_webhook_log já é inbox parcial · propor tabela onde já existe estrutura próxima · misturar sentinelas de cenários diferentes.

Regra para IA:
Nunca transformar hipótese diagnóstica em premissa de patch.
Antes de corrigir, provar se o evento está ausente, salvo com outro ID, soft-deleted, filtrado pela view/API/UI ou apenas mal classificado.

1. Onde está 100% correto

1.1 "Patch técnico não substitui estado operacional"

Patch corrige fluxo específico, não garante controle operacional sobre todos os eventos.

Ex: Patch A corrigiu outbound fromMe=true + @lid + conversa existente · não garante inbound fromMe=false + @lid · não garante recuperação de antigos · não garante que conversa subiu na fila.

1.2 A frente é dupla: ingestão + fila ordenada

Ingestão: a mensagem virou wa_message? Webhook passou pelo pipeline? Evento teve estado final?

Fila ordenada: a conversa subiu? last_message_at correto? last_message_text correto? Banco/view/API/UI concordam?

Uma mensagem pode estar salva e a fila ainda errada. Conversa pode estar no banco e a UI não atualizar.

1.3 Match: número > remote_jid > texto/horário > nome

  1. phone_e164 / número real.
  2. remote_jid / lid / s.whatsapp.net.
  3. provider_msg_id / wa_message_id.
  4. conversation_id + texto + janela de horário.
  5. display_name apenas como apoio.

Nome nunca deve ser chave canônica. "sandra" vs "Sandra" · "ANA FRATTI" vs "Ana" · "Região 22" · "Deus no controle" · número puro como display_name.

1.4 Não usar created_at como ordenação primária

Pós-Patch Timestamp A: sent_at = horário real WhatsApp · created_at = gravação/processamento.

Pré-Patch: sent_at e created_at próximos do horário do webhook · não horário real.

Regra: sempre separar análise pré-patch e pós-patch.

1.5 "Não fazer patch ainda"

Patch agora seria chute. Ainda precisa separar: (a) perda real · (b) outro ID · (c) soft-deleted · (d) trace enganoso · (e) mídia · (f) view/API/UI · (g) last_message_at stale.

1.6 Estados finais obrigatórios

Anti-pattern: auth_ok + event_messages_upsert e depois nada. Todo evento precisa terminal state.

1.7 Identidade WhatsApp como entidade própria

Correção: a tabela wa_contact_identities JÁ existe. O problema não é criar — é popular consistentemente.

2. Refinamentos necessários

2.1 R1 — "Param em auth_ok/event_messages_upsert" pode ser artefato de agregação

A view agrega múltiplos hits do mesmo provider_msg_id. Um pid pode ter: messages.upsert + messages.update + redelivery + status update + duplicate.

Risco: a IA olha linha agregada e conclui "parou em auth_ok". Mas pode ser que hit 1 salvou e hit 2 foi update/status que parou cedo de propósito.

Conclusão "parou cedo" só vale se query hit-por-hit provar que TODOS os hits relevantes não avançaram.

Correção do bucket: chamar de "suspeito de parada precoce" até auditoria hit-por-hit.

2.2 R2 — Match alternativo deve preceder "perda real"

Ordem obrigatória: provider_msg_id → wa_message_id → conv+texto exato → texto parcial → janela horário → soft delete → só então perda real.

Mesma mensagem semântica pode aparecer com IDs diferentes: Cloud salva wamid... · Evolution ecoa 3EB... · UI cria e atualiza provider_msg_id depois · webhook eco antes de updateStatus · reprocessamento.

2.3 R3 — created_at não é universal

2.4 R4 — saveOutbound sincroniza last_message_at apenas para sender='humano'

Pula sender='lara' e sender='system'. Intencional pra evitar auto-greeting subir conversa. Mas cria risco: mensagem legítima da Lara/IA/sistema pode não atualizar a fila.

2.5 R5 — wa_webhook_log já é event inbox semi-durável

Falta: terminal_state, normalized_event_id, retry_count, last_reprocess_at, reprocess_status, correlação robusta com wa_messages, job de reprocessamento, views de pendência.

Solução incremental:

Solução robusta futura: separar wa_event_inbox como tabela semântica · deixar wa_webhook_log como audit bruto.

4. Crítica à arquitetura proposta

4.1 C1 — State machine via wa_webhook_log antes de wa_event_inbox

6. Respostas diretas às duas perguntas implícitas

6.1 "Poderia ter sido evitado?"

Sim, em parte.

Poderia: perda silenciosa · ausência de terminal state · falta de pending queue · ausência de reprocessamento · dependência frágil de phone/remote_jid · fila stale sem alerta · dificuldade de provar se algo foi salvo.

O objetivo não é impedir todo evento estranho. O objetivo é impedir que evento estranho desapareça sem classificação.

6.2 "Patch pontual ou refatoração arquitetural?"

Os dois, em ordem:

  1. Patches cirúrgicos para perda operacional real já identificada.
  2. State machine MVP sobre wa_webhook_log.
  3. Reprocessador idempotente.
  4. População consistente de wa_contact_identities.
  5. Pending queue para LID sem mapping.
  6. Só depois considerar event inbox separado / refator maior.
Não usar "arquitetura ideal" como desculpa para não corrigir bug real.
Não usar "patch rápido" como desculpa para continuar sem auditabilidade.
