## Riscos antes da correção

1. **`b2b_voucher_dispatch_queue` com policy aberta** · qualquer authenticated user vê fila de outras clínicas (no futuro multi-tenant). Hoje single-tenant Mirian: dano = 0, mas é débito que vira P0 quando segunda clínica entrar.
2. **`webhook_processing_queue` com policy aberta** · idem · payloads de webhook contêm phone+payload sensível · authenticated NÃO deveria sequer enxergar (UI nem usa essa tabela direto).
3. **`mira_conversation_state` com policy aberta** · sem clinic_id, mas state_value é jsonb com possíveis dados sensíveis (voucher confirms, cp_step). authenticated NÃO deveria ler.

## Correções aplicadas (mig 800-14)

1. `mira_conversation_state` · DROP policy aberta · REVOKE pra anon/authenticated · service_role only
2. `b2b_voucher_dispatch_queue` · DROP policy aberta · 4 policies tenant-scoped + REVOKE anon
3. `webhook_processing_queue` · DROP policy aberta · 4 policies tenant-scoped + REVOKE anon
4. `inbox_notifications` · REVOKE anon explícito (defense-in-depth)
5. `_ai_budget` · REVOKE anon explícito (defense-in-depth)

## Sanity check pós-migration

Rode no SQL Editor do Supabase após aplicar a 800-14:

```sql
-- Não pode retornar nenhuma linha
SELECT
  pg_tables.tablename,
  pg_policies.policyname,
  pg_policies.qual,
  pg_policies.with_check
FROM pg_policies
JOIN pg_tables ON pg_tables.schemaname='public' AND pg_tables.tablename = pg_policies.tablename
WHERE pg_tables.schemaname = 'public'
  AND pg_tables.tablename IN (
    SELECT t.tablename FROM pg_tables t
     WHERE t.schemaname='public'
       AND EXISTS (
         SELECT 1 FROM information_schema.columns c
          WHERE c.table_schema='public'
            AND c.table_name=t.tablename
            AND c.column_name='clinic_id'
       )
  )
  AND (pg_policies.qual = 'true' OR pg_policies.with_check = 'true');
```
