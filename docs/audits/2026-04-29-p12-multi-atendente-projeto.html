Projeto · P-12 multi-atendente

Quem está vendo, quem assumiu, quem está digitando

P-12 era a última fraqueza estrutural pendente do roadmap /conversas (depois de P-13/P-14/P-15 viraram polish). Não dá pra fazer "rapidão" — é arquitetural: presença em tempo real, lock cooperativo, assignment explícito, indicador de digitação. Este doc é o projeto antes de codar.

Data: 2026-04-29 App: apps/lara Rota: /conversas Esforço estimado: L (large) Status: Projeto · não iniciado

O problema em 1 parágrafo

Hoje, se Mirian e Beatriz abrirem a mesma conversa simultaneamente, ambas digitam a mesma resposta sem saber. Mirian envia, Beatriz envia segundos depois — o paciente recebe duas respostas (às vezes contraditórias). Não há indicador de presença, não há lock, não há assignment ("este lead é seu"). É o gap que separa um inbox de uma central de atendimento real.

Sintomas observáveis em prod: respostas duplicadas, transferências silenciosas (Mirian assume e Beatriz não percebe), comentário "quem tá com este aqui?" no grupo do WhatsApp da equipe (que vira o canal de coordenação por falta de UI).

O escopo · 4 capacidades, 1 sistema

1 · Presença

Quem está logado em /conversas agora. Avatar no canto da página + lista lateral de online. Heartbeat 30s.

2 · Visualização

"Beatriz está vendo essa conversa" embaixo do header do MessageArea. Heartbeat por conversa aberta.

3 · Digitação

"Mirian está digitando..." quando textarea da conversa tem foco e conteúdo. Throttle 2s + cleanup ao parar.

4 · Assignment

"Atribuído a Beatriz" no painel direito. Dropdown com membros ativos da clínica. Persistido em wa_conversations.assigned_to uuid.

Decisão arquitetural · transporte da presença

3 caminhos viáveis. Cada um troca complexidade por flexibilidade.

Caminho 1 · Supabase Realtime Presence

Recomendado

Como funciona: Supabase já tem channel('conversa-{id}').track({user, status}). Cada cliente entra no canal, server distribui presence diff, todos os clientes recebem snapshot atualizado. Builtin no SDK que já usamos.

Pro: zero infra nova · auto-cleanup em disconnect · multi-tenant safe (filtra por clinic_id no canal).
Contra: pricing por concurrent connections em planos altos · debug menos visível que custom.

Caminho 2 · Postgres NOTIFY/LISTEN

Como funciona: trigger em wa_conversations dispara NOTIFY · cliente conecta via Supabase Realtime postgres_changes que já usamos pro SSE.

Pro: usa o que já temos · sem terceira pilha.
Contra: precisa coluna active_viewers jsonb com TTL · tem que GC manual · presença binária (não captura "digitando").

Caminho 3 · WebSocket próprio

Como funciona: servidor Node separado pra presence + typing · clientes conectam direto.

Pro: controle total · custom protocol.
Contra: infra nova · deploy separado · scaling próprio. Overkill pro caso.

Escolha: Caminho 1 (Supabase Realtime Presence). Já temos o SDK + canal SSE rodando. Custo zero adicional, ~30 linhas de código por hook. Se Mirian crescer o time pra 20+ atendentes simultâneos e o plano Supabase pesar, migramos pra Caminho 3.

Schema · migration 87

wa_conversations · 2 colunas novas

ALTER TABLE public.wa_conversations
  ADD COLUMN IF NOT EXISTS assigned_to uuid REFERENCES auth.users(id),
  ADD COLUMN IF NOT EXISTS assigned_at timestamptz;

CREATE INDEX IF NOT EXISTS wa_conversations_assigned_to_idx
  ON public.wa_conversations(assigned_to)
  WHERE assigned_to IS NOT NULL;

NOTIFY pgrst, 'reload schema';

Presença/digitação NÃO precisam de tabela — vivem em memória no canal Realtime. Apenas assigned_to (estado durável) vai pro DB.

RPC · assignment seguro

CREATE OR REPLACE FUNCTION public.wa_conversation_assign(
  p_conversation_id uuid,
  p_user_id uuid
) RETURNS jsonb
LANGUAGE plpgsql SECURITY DEFINER
SET search_path TO 'public', 'pg_temp'
AS $$
BEGIN
  -- Auth: apenas membros da mesma clinic_id podem ser assignados
  IF NOT EXISTS (
    SELECT 1 FROM profiles p
    JOIN wa_conversations c ON c.clinic_id = p.clinic_id
    WHERE c.id = p_conversation_id AND p.id = p_user_id
  ) THEN
    RETURN jsonb_build_object('ok', false, 'error', 'user_not_in_clinic');
  END IF;

  UPDATE wa_conversations
    SET assigned_to = p_user_id, assigned_at = now()
    WHERE id = p_conversation_id
      AND clinic_id = public.app_clinic_id();

  RETURN jsonb_build_object('ok', true);
END $$;

GRANT EXECUTE ON FUNCTION public.wa_conversation_assign(uuid, uuid) TO authenticated;

API · endpoints

MétodoRotaFunção
POST /api/conversations/[id]/assign Body { user_id } → chama RPC wa_conversation_assign. Resposta { ok, assigned_to, assigned_at }.
DELETE /api/conversations/[id]/assign Remove assignment (set null).
GET /api/clinic/members Lista usuários ativos da clínica (id, nome, role) pra dropdown de assign.

Presença/digitação não têm endpoint REST. São WS via Supabase Realtime canal presence:clinic-{clinic_id}:conversa-{conv_id}.

1 · usePresence(scope)

  channelKey: `clinic-${clinicId}:${scope}`,
  user: { id, name, avatar_url },
});

// onlineUsers: Array<{ id, name, avatar_url, typing: boolean, joined_at }>
// sendTyping(true|false): broadcast typing state, throttled 2s

Wrapper sobre supabase.channel(...).track(...). Heartbeat automático. Cleanup em unmount.

UI · 3 surfaces

1 · Inbox-level presence (sidebar header)

Linha discreta abaixo do título "Conversas" mostrando ● Você · ● Beatriz · ● Mirian com avatares pequenos (24px) e dot verde de presença. Click no avatar abre tooltip com nome + "online há X min".

2 · Conversation viewers (MessageArea header)

Abaixo do nome do paciente, linha minúscula em cinza: "Beatriz está vendo · Mirian digitando...". Aparece só quando há outros viewers além de você. Pulsa quando alguém está digitando.

3 · Assignment (LeadInfoPanel)

Section nova acima de "Etapa Atual" — "Atribuído a" com:

  • Sem assignment: botão + Atribuir abre dropdown de membros
  • Atribuído a você: avatar + nome + botão Liberar
  • Atribuído a outro: avatar + nome + texto "Beatriz está cuidando" + botão Assumir mesmo assim (warning amarelo)

Soft-lock — não bloqueia o envio, só sinaliza. Se Mirian quiser sobrescrever, pode (com warning). Hard-lock vira ressentimento operacional.

Riscos & edge cases

Alto
Dois atendentes assignam ao mesmo tempo · race condition. RPC já tem UPDATE atômico. Resposta retorna estado final · UI reconcilia.
RPC atomic · no risk real após mig.
Alto
Atendente fecha laptop sem logout · presença fica zumbi até WebSocket timeout (~60s no Supabase). Outras pessoas veem ele "online" minutos após sair de casa.
Heartbeat 30s · mostra "ativo há 2 min" se >60s sem ping. Aceitável.
Médio
Custo Supabase Realtime · plano Pro inclui 200 connections simultâneas. Mirian tem 2-3 atendentes hoje, ok. Se crescer pra 20+, vira orçamento.
Monitorar · revisitar quando mudar de plano.
Médio
Typing indicator gera ruído · todo mundo vendo todo mundo digitando vira distração. Deveria mostrar só na conversation-level (não inbox).
Decisão: typing só na conversa aberta, nunca na sidebar.
Baixo
Soft-lock vira disputa · "Mirian assumiu mesmo assim mesmo tendo Beatriz" — pode gerar atrito político.
Auditar via timeline (já implementado em SA-07) · supervisão revê.
Baixo
Multi-tenant leak · dois clinics no mesmo Supabase, presença vaza canal.
Channel key inclui clinic-{clinic_id}. Auditar.

Contratos canônicos reafirmados

  • ADR-012 · Repository pattern: ProfileRepository.listByClinic, ConversationRepository.assign — todo DB via repos
  • ADR-028 · Multi-tenant: clinic_id sempre via loadServerContext() JWT, channel keys incluem clinic prefix
  • Stack inegociável: Next.js 16 · React 19 · Tailwind 4 · sem novas deps · Supabase SDK já instalado
  • Visual: Cormorant Garamond + champagne · cores semânticas (verde online, amarelo warning, cinza idle)
  • Sem hard-lock: soft-lock + auditoria. Confiança > controle.
  • Migration idempotente: CREATE INDEX IF NOT EXISTS, ADD COLUMN IF NOT EXISTS, NOTIFY pgrst
